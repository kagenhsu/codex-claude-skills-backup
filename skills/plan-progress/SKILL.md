# plan-progress｜專案進度自動偵測

## 何時使用

- 接手一個放置一段時間的專案，想 30 秒內知道做到哪了。
- 換電腦 / 換 AI 接手前後對齊進度，避免「上次做到哪」翻聊天紀錄。
- 雙刀流 / 單刀流交棒時，先跑一次當作交接憑據。
- 想用一句話回答「下一步該幹嘛」。

## 前置條件

專案根目錄需有 `專案規劃表.md`，格式建議：

```
## 第 N 階段｜階段名稱
### 這一階段要做什麼
- ...
### 細項開發步驟
- N.M 小步驟描述（含關鍵檔名／路徑暗示，例如 `data/skills.yaml`、`install.sh`）
```

只要每個小步驟的第一個字串符合 `\d+\.\d+`（例如 `4.3`），且步驟描述包含可辨識的檔名／路徑，就能推論規則。

## 流程（AI 接到觸發句後依序執行）

1. **讀規劃表**：開啟 `專案規劃表.md`，以 `## ` 切階段，從每階段抽出含 `N.M` 編號的小步驟。
2. **檢查規則快取**：
   - 若根目錄有 `.plan-rules.yaml` → 直接讀取，跳到第 4 步。
   - 沒有 → 進入第 3 步。
3. **推論規則並寫快取**：對每個有編號的小步驟，閱讀步驟內文，從反引號（`` ` ``）字串、檔名、路徑提示中推論「哪個檔案存在能證明這步完成」。產出規則寫進 `.plan-rules.yaml`，格式：
   ```yaml
   # 每個 step id 對應一個或多個 glob／regex 模式
   "1.6":
     - AGENTS.md
   "3.7":
     - data/prompts.yaml
   "6.3":
     - codex-skills-backup.tar.gz
   ```
4. **列出專案檔案**：用 `find . -type f`（跳過 `.git/`、`node_modules/`、`__pycache__/`、`.codex-tmp/`、`.DS_Store`）取得相對路徑清單。
5. **逐條套規則**：對每個小步驟，看至少一條規則能否在檔案清單中命中；命中即標記為完成。
6. **計算結果**：每階段完成率 + 找出第一個未完成的小步驟（即「下一步」）+ 總百分比。
7. **輸出報告**（見下方格式）。

## 輸出格式

```
📋 plan-progress 報告

專案：{root name}
目前階段：第 X 階段｜{title}
下一個小步驟：X.Y｜{描述}
總進度：N / Total（{pct}%）

各階段進度
- 第 0 階段｜⋯ ✓ 已完成 6/6
- 第 1 階段｜⋯ ✓ 已完成 8/8
- 第 2 階段｜⋯ ⏳ 進行中 5/9（目前在這裡）
- 第 3 階段｜⋯ ⭕ 尚未開始 0/11
- ⋯

下一步建議
{依「下一個小步驟」的描述，給出一段最具體的下一步行動}
```

如使用者只要簡短摘要，可僅輸出前 4 行（專案／目前階段／下一個小步驟／總進度）。

## 觸發句

- 用 plan-progress 看這個專案做到哪了
- 跑 plan-progress 報目前進度
- 用 plan-progress 告訴我下一步要做什麼
- plan-progress：產出進度報告

## 不做的事

- **不修改任何專案原始檔**。唯一會寫入的檔案是 `.plan-rules.yaml`（規則快取），且僅在不存在時才寫一次。
- 不執行 `git` 任何寫操作（不 commit、不 push、不切分支）。
- 不安裝套件、不呼叫網路。
- 不刪除 `.plan-rules.yaml`，除非使用者明確要求重推規則。

## 規則快取重建

使用者說「重新推論 plan-progress 規則」或刪除 `.plan-rules.yaml` 後再跑，會重新進入第 3 步。

## 與本控制台的關係

控制台「專案開發（雙刀／單刀）→ 第 4 步：看開發階段」最下方那張「本專案的全部開發階段」卡片，就是把這個 skill 的偵測邏輯做成網頁版（內建一組偵測規則，不需要 `.plan-rules.yaml`）。這個 skill 是 CLI 版本，適合：

- 不想開瀏覽器，直接在終端機問 AI 進度。
- 處理任何專案（不只本控制台），靠 AI 自動推論規則。
- 想把進度結果寫進其他文件、Slack、或自動化流程。

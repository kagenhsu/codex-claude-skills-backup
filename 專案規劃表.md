# 專案規劃表

> 專案：Codex / Claude Skills 管理系統＋提示詞庫（Skill 助手控制台）
> 版本：v1.1｜建立日期：2026-06-16
> 目的：把整個專案從零開始，拆成新手也能照著走的「一階段一個小目標」，每個階段再切成可逐步打勾的細項。

---

## 怎麼看這份表

每個階段都會回答四件事：

1. **這一階段要做什麼**：這個階段的總體方向。
2. **細項開發步驟**：把總體方向再切成可一條一條打勾的小任務，照順序做。
3. **會產出什麼**：做完之後手上會多出哪些檔案、文件或結果。
4. **完成後怎麼判斷可以進下一步**：用「能不能做到 ⋯⋯」的方式檢查，能就過關，不能就回去補。

不要急著跳階段。前一階段的「產出」是下一階段的「材料」，少一樣後面就會卡住。

## 第 0 階段｜準備與啟動

### 這一階段要做什麼

- 想清楚這個專案是「給誰用」、「解決什麼痛」。本案的對象是已經裝了一堆 Codex / Claude skills、但記不住觸發句的自己與同事。
- 確認自己的角色：你是專案負責人，AI（Claude／Codex）是執行者。你下決策、AI 動手。
- 確認手邊有什麼可以重用：既有的 `codex-skills-backup.tar.gz`、已安裝的 45 個 skills、`AGENTS.md` 規則文件。
- 設定一條工作節奏：「一個對話只做一件事，做完總結再開下一件」。

### 細項開發步驟

- **0.1 一句話寫下「為什麼做」**：用「給___用，解決___問題」的格式寫一句話。例：「給自己與同事用，解決 skill 太多記不住觸發句、複製貼上要翻舊對話的問題」。
- **0.2 清點現有資源**：條列出已經存在、可以直接重用的東西——已安裝的 skills 數量、備份壓縮檔、可用的 AI 工具（Claude Code、Codex CLI）、可用的開發環境（mac / Python / 瀏覽器）。
- **0.3 寫下「本期不做的事」**：強迫自己排除誘惑。本案排除：不做雲端服務、不做帳號系統、不自動安裝 skill、不做 CLI 內查詢。
- **0.4 確認角色分工**：寫下「人做決策、AI 做執行」的分工。決策包含：要不要做、做哪一版、上不上架；執行包含：寫程式、寫文件、改檔案。
- **0.5 設定工作節奏**：定一條鐵律「一個對話只做一個明確任務」，並寫進 AGENTS.md。
- **0.6 建立工作資料夾**：選一個固定位置放專案（本案：`~/Documents/codex skills管理/`），避免散落在桌面或下載夾。

### 會產出什麼

- 一段「為什麼要做這個專案」的白話描述（兩三句就好，存進腦袋或便利貼）。
- 一份「目前手上已經有什麼」的清點清單。
- 一個固定的工作資料夾位置。

### 完成後怎麼判斷可以進下一步

- 能用一句話對外人說明「這個專案做完之後，誰會比較輕鬆，輕鬆在哪裡」。
- 能說出「最不能砍掉的功能是哪一個」——本案是「skill 觸發句一鍵複製」。
- 能說出「本期不做什麼」——本案是不連網、不自動安裝、不做 CLI 內查詢。

## 第 1 階段｜需求整理（PRD）

### 這一階段要做什麼

- 把痛點、目標、功能、不做的事、驗收標準寫進一份 PRD。
- 列功能清單時，每一條都要對應到一個痛點，沒對應到的就先不做。
- 把「驗收標準」寫成可以實際操作的句子，例如「雙擊 index.html，搜尋『翻譯』能找到 baoyu-translate」，而不是「介面好用」。

### 細項開發步驟

- **1.1 痛點清單**：先寫 3–5 個你自己最常遇到的痛點。例如「裝了但忘了用」「觸發句忘了」「常用提示詞每次重打」「不知道新 skill 安不安全」。
- **1.2 寫專案目標**：對應每個痛點寫一句目標。例如「不用記觸發句，點按鈕就能複製」。
- **1.3 功能清單與痛點對照**：做一張表「功能 ↔ 對應痛點」，沒對應到痛點的功能砍掉。
- **1.4 寫「不做的事」**：把第 0 階段定的範圍正式寫進 PRD 第 5 節，避免做著做著偷加功能。
- **1.5 寫可操作的驗收標準**：每條都用「動作 + 預期結果」格式。例：「雙擊 index.html → 搜尋『翻譯』 → 看到 baoyu-translate 卡片」。
- **1.6 寫 AGENTS.md 協作規則**：列 5–8 條 AI 必須遵守的規矩，例如「先看 PRD 再動手」「不刪檔」「不確定先問」「修改後總結改了什麼」。
- **1.7 PRD 自審**：把 PRD 念給沒參與的人聽，或隔半天自己重看一次，看會不會有「這條到底是什麼意思」的卡頓。
- **1.8 PRD 定版**：標上版本號（v1.0）與日期，正式進入開發。

### 會產出什麼

- `PRD.md`：含背景、目標、痛點、功能清單、不做的事、驗收標準。
- `AGENTS.md`：寫給 AI 看的協作規則（一次只做一件事、不刪檔、不確定先問）。
- 一份初步的功能清單（本案 6 項：skill 目錄、提示詞庫、全域搜尋、風險標示、安檢 SOP、資料維護方式）。

### 完成後怎麼判斷可以進下一步

- PRD 裡每一條功能都能回答「砍掉它會痛嗎？」，會痛才留。
- 驗收標準至少有一條是「可以親手點點看、複製貼上看」的具體動作。
- 把 PRD 念給一個沒參與的人聽，對方能複述出你要做的東西大概長什麼樣子。

## 第 2 階段｜環境架設

### 這一階段要做什麼

- 決定使用哪些工具：本案用 macOS 終端機、VS Code、Python 3（跑 build.py）、瀏覽器（看 index.html）。
- 建好工作目錄結構，初始化 Git，先做一次 commit 當作起點。
- 設定 `.gitignore`：不要把暫存檔、`.DS_Store`、`.codex-tmp/` 之類的東西版控進去。
- 確認 Python 與 PyYAML 之類最低需要的套件可以執行。

### 細項開發步驟

- **2.1 列技術棧**：寫下「會用到什麼」。本案：Python 3 + PyYAML（資料處理）、HTML/CSS/JS（控制台介面）、Bash + PowerShell（安裝腳本）、Git/GitHub（版控）、VS Code（編輯）。
- **2.2 建立資料夾骨架**：在工作目錄建出 `data/`、`scripts/`、`docs/`、`skills/`，每個資料夾放一個 `.gitkeep` 讓 Git 認得。
- **2.3 初始化 Git**：`git init` → 建 main 分支 → 寫第一個 commit 訊息「chore: init project skeleton」。
- **2.4 寫 .gitignore**：把 `.DS_Store`、`.codex-tmp/`、`*.pyc`、`__pycache__/`、`.vscode/`、IDE 暫存檔加進去。
- **2.5 確認 Python 版本**：`python3 --version`，本案需 3.8 以上；確認 `pip install pyyaml` 能裝起來。
- **2.6 寫 Hello World 版 index.html**：只放一行 `<h1>Hello</h1>`，雙擊能在瀏覽器打開。
- **2.7 寫 Hello World 版 build.py**：只做一件事——印出「build ok」，確認 Python 能執行專案內腳本。
- **2.8 建立遠端 repo**：在 GitHub 建一個空 repo（本案 `kagenhsu/codex-claude-skills-backup`），把本地 push 上去。
- **2.9 環境檢查單**：寫一張小筆記，紀錄「Python 版本、PyYAML 版本、Git 版本、Node 版本（若用到）」，方便換電腦時對齊。

### 會產出什麼

- 初始化好的 Git repo（本地或 GitHub 上的 `codex-claude-skills-backup`）。
- 空白但結構正確的資料夾：`data/`、`scripts/`、`docs/`、`skills/`。
- 一份「環境檢查通過」的小筆記：Python 版本、可以執行 `python scripts/build.py` 不報錯。

### 完成後怎麼判斷可以進下一步

- 能在終端機進到專案資料夾，跑 `git status` 不會報錯，看到的就是預期的乾淨狀態。
- 能跑一次空版的 `scripts/build.py`（就算還沒功能），不會因為「找不到 Python」「缺套件」失敗。
- 雙擊一個最簡單的 `index.html`（裡面只寫 Hello），瀏覽器能正常打開。

## 第 3 階段｜基礎系統（資料與骨架）

### 這一階段要做什麼

- 設計資料格式：本案決定用 YAML，因為人類好讀、AI 也好寫。
- 把 45 個 skill 的中文說明、用途、風險、1–3 句觸發句，先列進 `data/skills.yaml`。
- 把常用提示詞分 5 大類（專案啟動／開發過程／版本控制／整合串接／上線部署＋安全檢查），寫進 `data/prompts.yaml`。
- 寫 `scripts/build.py`：讀 YAML → 套版型 → 產出 `index.html`。版型先用最樸素的 HTML，能顯示就好。
- 確認「資料來源唯一」：以後改內容只改 YAML，不直接改 HTML，避免兩邊不同步。

### 細項開發步驟

- **3.1 設計 skill YAML schema**：先決定每個 skill 卡片要顯示什麼欄位——`name`、`category`、`summary`、`triggers`、`notes`、`risk_level`。
- **3.2 設計 prompt YAML schema**：欄位包含 `category`、`title`、`content`、`tags`。
- **3.3 萃取現有 skills 清單**：從 `~/.codex/skills` 與 `~/.claude/skills` 列出所有 skill 名稱，做一份完整名單（本案 45 個）。
- **3.4 補中文說明**：對每個 skill 填上中文用途、適用情境、注意事項，可請 AI 一批一批產初稿，你逐個確認。
- **3.5 補觸發句**：每個 skill 寫 1–3 句最常用的觸發句，例如「用 baoyu-translate 把這篇文章翻成繁體中文」。
- **3.6 標風險等級**：`baoyu-danger-*` 系列標 `high`，會連網或寫檔的標 `medium`，純說明類標 `low`。
- **3.7 寫 prompts.yaml**：把 5 大類提示詞（專案啟動／開發過程／版本控制／整合串接／上線部署＋安全檢查）逐條輸入，每張卡 50–200 字。
- **3.8 寫 build.py 主流程**：① 讀 `data/*.yaml` → ② 用 Python 字串模板組 HTML → ③ 輸出到 `index.html`。
- **3.9 設計最簡 HTML 模板**：頭部 `<h1>` 標題、分區 `<h2>`、每個卡片 `<article>`。先不寫 CSS，能看就好。
- **3.10 第一次完整 build**：執行 `python scripts/build.py`，雙擊 `index.html`，確認 45 個 skill + 提示詞全顯示。
- **3.11 驗證單一資料源**：故意把某個 skill 的 summary 改成「測試」，重新 build，看 HTML 是不是真的變了；確認沒有人會手動去改 HTML。

### 會產出什麼

- `data/skills.yaml`、`data/prompts.yaml`（資料源頭）。
- `scripts/build.py`（產生器）。
- `index.html`（產出物，能跑但還很陽春）。
- 一份簡單規則：「日常維護 = 改 YAML → 跑 build → 重新打開 HTML」。

### 完成後怎麼判斷可以進下一步

- 改 `data/skills.yaml` 任一個 skill 的說明文字，跑 `build.py`，重新打開 `index.html` 後該段文字確實有變。
- 45 個 skill 全部都列得出來，沒有漏掉或重複。
- 控制台首頁能看到所有 skill 和提示詞的卡片，雖然醜，但資料完整。

## 第 4 階段｜功能開發（讓控制台真的好用）

### 這一階段要做什麼

把基礎骨架升級成 PRD 列出的 6 個功能：

1. **Skill 目錄**：每張卡片有中文用途、適用情境、注意事項、觸發句。
2. **觸發句一鍵複製**：按按鈕，內容進剪貼簿，貼到 Codex / Claude CLI 就會跑。
3. **提示詞庫**：5 大分類、約 15 張卡片，同樣一鍵複製。
4. **全域搜尋**：上方搜尋列輸入關鍵字，skill 與 prompt 即時過濾。
5. **風險標示**：每個 skill 標 低／中／高 風險，`baoyu-danger-*` 系列要紅色標高風險。
6. **安檢 SOP 頁籤**：寫好「裝新 skill 前要看的紅旗清單」與「安檢提示詞」。

開發節奏建議：一次只開一個功能，做完手動點一次，確定沒壞掉再開下一個。

### 細項開發步驟

- **4.1 Skill 卡片版型**：在 `build.py` 模板裡寫好卡片結構：標題、分類標籤、用途、適用情境、注意事項、觸發句區。先不加 CSS。
- **4.2 加基本 CSS**：寫一份淺色主題 CSS（卡片、間距、字體），讓畫面從「文件」變成「卡片牆」。
- **4.3 觸發句複製按鈕**：每句觸發句加一個「複製」按鈕，用 `navigator.clipboard.writeText()` 把該句內容寫到剪貼簿。
- **4.4 複製成功提示**：按下複製後，按鈕短暫顯示「已複製 ✓」，2 秒後變回原文字。
- **4.5 分類側欄**：左側列出 skill 分類（內容處理、開發協作、影像生成、平台發佈⋯⋯），點任一分類只顯示該分類卡片。
- **4.6 提示詞頁籤**：在頂部加分頁「Skill 目錄 / 提示詞庫 / 安檢 SOP」，三頁切換。
- **4.7 提示詞卡片**：套用同樣的卡片版型 + 一鍵複製按鈕。
- **4.8 全域搜尋列**：上方放一個 input，輸入時即時用 JS 過濾顯示的卡片（同時搜 skill 與 prompt 名稱、summary、tags）。
- **4.9 搜尋空狀態處理**：搜不到結果時顯示「沒有符合的 skill 或提示詞」，不要讓畫面整片空白。
- **4.10 風險標示**：每張 skill 卡片右上角加色塊（綠 / 黃 / 紅），對應 `risk_level` 欄位。
- **4.11 高風險警語**：`risk_level: high` 的卡片在「複製按鈕」上方多一行紅字警告：「此 skill 可能寫檔／連網／呼叫外部服務，請確認後再用」。
- **4.12 安檢 SOP 頁籤內容**：寫三段——「裝新 skill 前要做的 5 件事」「紅旗清單（可疑的訊號）」「安檢提示詞（給 Claude 跑審查的範本）」。
- **4.13 響應式調整**：在筆電與手機螢幕各看一次，卡片不要擠在一起或跑版。
- **4.14 把開發紀錄寫進 `docs/skill-console-plan.md`**：記下每個功能為什麼這樣做、做了什麼權衡。
- **4.15 列出已知小毛病**：開一個 `docs/v1-1-plan.md` 或 issue 清單，把暫時不修但日後要修的事寫下來。

### 會產出什麼

- 升級後的 `index.html`，含分類側欄、搜尋列、複製按鈕、風險色塊、安檢頁籤。
- `docs/skill-console-plan.md`：把每個功能怎麼長、為什麼這樣做寫下來，未來自己回看也懂。
- 一份「已知小毛病清單」：例如某些 skill 觸發句還沒寫好、某些分類圖示還沒換，留到下一階段修。

### 完成後怎麼判斷可以進下一步

- 6 個功能逐項手動測過，每一項都能用「我做了什麼動作 → 看到什麼結果」描述。
- 搜尋「翻譯」確實能找到 `baoyu-translate`；搜尋「安全」能跳到安檢相關內容。
- 點任一複製按鈕，貼到記事本，內容跟卡片上看到的一字不差。
- `baoyu-danger-*` 系列卡片顯示成紅色高風險，不會跟一般 skill 混在一起。

## 第 5 階段｜驗收（自己當第一個使用者）

### 這一階段要做什麼

- 拿 PRD 的「驗收標準」一條一條對：能做到就打勾，做不到就回第 4 階段補。
- 找一個「沒參與開發」的同事（或自己隔一天再看）試用：能不能在不問人的情況下，從打開 index.html 到把觸發句貼進 CLI 跑起來。
- 把試用過程中聽到的「卡住的地方」記下來，分成：必修（影響能不能用）、可修（影響好不好用）、之後再說（nice to have）。
- 必修的修完，再驗一次。

### 細項開發步驟

- **5.1 印出 PRD 驗收清單**：把 PRD 第 6 節的驗收標準逐條抄出來，做一份可打勾的 checklist。
- **5.2 自我驗收**：自己照清單操作一次，每條打勾或標記失敗原因。
- **5.3 隔日重看**：故意過一晚再開一次 `index.html`，從一個「半陌生使用者」的角度看哪裡卡。
- **5.4 找一位真正的盲測者**：找一位完全沒看過這個專案的同事，給他 README，看他能不能自己裝起來、用起來。
- **5.5 觀察與紀錄**：盲測時不要插話，只記錄「他在哪一步卡住、用了幾分鐘、有沒有問問題」。
- **5.6 回饋分類**：把所有問題分成三類——必修（壞掉、誤導）、可修（不順但能用）、之後再說（功能增強）。
- **5.7 修必修項**：回到第 4 階段修，修完重新 build。
- **5.8 第二次驗收**：用同樣的 checklist 再跑一次，確認必修項真的解決，且沒有改壞其他地方。

## 附錄 A：日常維護的迷你循環

上架之後，這個專案不是「做完就放著」，而是會持續長大。建議的日常循環：

1. 看到好用的 skill 或提示詞 → 收進對應的 `data/*.yaml`。
2. 跑 `python scripts/build.py` → 重新產出 `index.html`。
3. 雙擊打開 `index.html` 確認新內容有出現、沒有壞掉舊內容。
4. `git commit` + `git push`，同事下次更新就會拿到。

如果懶得自己改，直接把新內容丟給 Claude，請它依照 `AGENTS.md` 的規則代為更新即可。

## 附錄 B：細項任務追蹤建議

- 每個階段的細項步驟（例如 4.3、4.4），都可以視為一個獨立的 commit 或一個 AI 對話的範圍。
- 建議命名 commit 用「階段碼.細項碼 + 簡述」，例如：`4.3 add copy-to-clipboard button on triggers`。
- 一次只動一個細項，做完就 commit，避免一改一大堆東西混在一起，事後難回溯。
- 卡住時，回到該細項的「要做什麼」描述，確認自己有沒有偏題；如果發現該細項其實還能再拆，就再拆成 a / b / c 三步，逐步消化。

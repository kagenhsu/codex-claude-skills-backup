# v1.1 規劃：快速看板與組合包

版本：v1.1 草案（基於 v1 commit `7efccbb`）
狀態：第 1 階段 Codex 規劃，等待使用者確認

## 1. 背景

v1 已上線「收錄新內容」、「二刀流中控」靜態教學、「專案啟動」快捷篩選、「繁中 GitHub 上傳」與「跨地點 / 跨系統接續」等內容，目前控制台實測有 38 張 prompt 與二刀流中控 tab。

v1.1 要補兩個使用上的小痛點：

- 使用者打開控制台時，看不到目前 dual-ai-workflow 到哪一階段，還是要回去翻 `DUAL-AI-STATE.md`。
- 常用工作流常常要連續複製好幾張提示詞，例如「開新專案」、「審查修正」、「下班跨電腦接續」，點很多次會累。

所以 v1.1 先做兩個小功能：快速看板與常用組合包。

## 2. 功能群 V — DUAL-AI-STATE 快速看板

### 2.1 目標

在控制台新增一個「快速看板」區塊，讓使用者把 `DUAL-AI-STATE.md` 內容貼進來，控制台用純前端方式解析並顯示目前狀態。

這個功能只幫使用者「看懂目前進度」，不會修改任何檔案。

### 2.2 使用方式

1. 使用者打開控制台。
2. 到「二刀流中控」或未來指定位置找到「貼上 DUAL-AI-STATE.md」文字區。
3. 把 `DUAL-AI-STATE.md` 全文貼進去。
4. 控制台立刻解析並顯示：
   - 目前階段
   - 上一步做了什麼
   - 下一步要做什麼
   - 未解決問題
   - 最後更新時間
5. 五階段卡片會依目前階段做 highlight，讓使用者一眼知道現在在哪。

### 2.3 純前端解析

v1.1 不做後端、不讀取本機檔案，也不自動存取 `DUAL-AI-STATE.md`。

控制台只會解析使用者手動貼上的文字，解析方式先採用簡單規則：

- 找 `目前階段：` 後面的文字，判斷是第 1～5 階段、已完成，或無法判斷。
- 找 `已完成事項：` 區塊，摘要最後幾行當作「上一步」。
- 找 `下一步：` 區塊，顯示下一步內容。
- 找 `未解決問題：` 區塊，顯示問題列表。
- 找 `最後更新時間：` 後面的文字，顯示更新時間。

如果格式不完整，就只顯示能解析到的欄位；不要讓頁面壞掉。

### 2.4 錯誤與空白處理

如果使用者貼上空字串、亂碼、或不是狀態檔的內容：

- 不報錯。
- 顯示「無法解析，請確認貼上的是 DUAL-AI-STATE.md 全文」。
- 五階段卡片不 highlight，或顯示為「尚未判斷」。

### 2.5 LocalStorage 暫存

貼上的內容可用瀏覽器 LocalStorage 暫存，方便重新整理後不遺失。

建議 key：

- `workflow-state-draft`

重要提醒：

- 內容只存在使用者自己的瀏覽器。
- 不上傳。
- 不同步到 GitHub。
- 不寫回 `DUAL-AI-STATE.md`。

### 2.6 放置位置

v1.1 建議先放在「二刀流中控」tab 上方，因為使用者看階段時通常也會看五階段流程卡。

暫不新增獨立 tab，避免控制台頁籤過多。

## 3. 功能群 W — 常用組合包

### 3.1 目標

新增「組合包」概念：把常一起使用的多張 prompt 包成一組，使用者按一個 chip 就能複製整段交辦包。

小白版說明：以前要一張一張複製；v1.1 讓常用流程變成「一包帶走」。

### 3.2 新增資料檔

新增：

```text
data/combos.yaml
```

v1.1 建議先用「引用 prompt title」的 schema，不直接內嵌 prompt 全文。

原因：

- 避免同一段 prompt 在 `prompts.yaml` 和 `combos.yaml` 重複兩份。
- 原提示詞更新時，組合包會自動拿到新版內容。
- 比較不容易發生內容不同步。

草案 schema：

```yaml
- title: 開新專案起手包
  usage: 開始一個新專案時，快速建立規則、PRD 與狀態檔。
  category: 專案啟動
  steps:
    - prompt_title: 從討論結果產出 AGENTS.md（設計方向確認後用）
      custom_intro: 第一步，先把討論過的規則整理成 AGENTS.md。
    - prompt_title: 梳理需求並產出 PRD
    - prompt_title: 查目前進度（讀 DUAL-AI-STATE.md）
```

### 3.3 顯示方式

在「提示詞庫」tab 頂端新增「常用組合包」區塊或 chip 列。

每個組合包 chip 顯示：

- 組合包名稱
- 一句用途
- 包含幾張 prompt
- 「複製整包」按鈕

複製出的文字應該包含清楚分隔線，例如：

```text
以下是「開新專案起手包」。

--- Prompt 1：從討論結果產出 AGENTS.md（設計方向確認後用） ---
...

--- Prompt 2：梳理需求並產出 PRD ---
...
```

### 3.4 三組初始組合包

v1.1 先放三組，不一次做太多。

#### 3.4.1 開新專案起手包

用途：剛開始做新專案、已討論完方向，要建立規則與需求文件時使用。

建議包含：

- `從討論結果產出 AGENTS.md（設計方向確認後用）`
- `梳理需求並產出 PRD`
- `首次上傳 GitHub：繁體中文專案敘述`

#### 3.4.2 二刀流審查修正包

用途：Codex 做完實作後，要交給 Claude Code 審查，再回到 Codex 修正。

建議包含：

- `給 Codex 的逐條修正轉交提示詞（模板）`
- `給 Codex 的存檔收尾提示詞（模板）`
- `③ 第三階段：Claude Code（VS Code）審查`

#### 3.4.3 跨地點接續包

用途：公司、家裡、Mac mini、Windows 電腦之間接續同一個專案。

建議包含：

- `跨地點 / 跨系統接續`
- `查目前進度（讀 DUAL-AI-STATE.md）`
- `額度快滿自動存檔＋換手`

### 3.5 找不到 prompt 時的處理

如果 `combos.yaml` 引用的 `prompt_title` 在 `prompts.yaml` 找不到：

- 不讓整頁壞掉。
- 在組合包卡片顯示「缺少提示詞：某某標題」。
- 複製整包時略過缺少的 prompt，並在最前面列出缺漏提醒。

## 4. 不做的事

- 不上傳貼上內容。
- 不寫 `DUAL-AI-STATE.md`（控制台只讀，不寫）。
- 不做意圖快搜、Skill 同步檢查、版本紀錄區塊（留 v1.2）。
- 不做後端服務。
- 不自動讀本機檔案。
- 不自動 push GitHub。

## 5. 驗證方式

- 貼上一份有效 `DUAL-AI-STATE.md` → 五階段卡正確 highlight。
- 貼上空字串或亂碼 → 不報錯，顯示「無法解析」提示。
- 組合包 chip 點下去複製整段，貼回編輯器看分隔線正確。
- `data/combos.yaml` 引用不存在的 prompt title 時，頁面不壞並顯示缺漏提醒。
- 既有 7 個 tab 與搜尋未被打破。
- 搜尋「角色」「翻譯」「審查」「繁體」「跨地點」仍能命中既有內容。
- 執行 `python3 scripts/build.py` 成功重建 `index.html`。

## 6. 開放問題

1. `combos.yaml` 的 schema 要用 `title + steps[{prompt_title, custom_intro?}]`，還是直接內嵌 prompt 全文？
   - Codex 建議：v1.1 先用 `prompt_title` 引用，避免 prompt 內容重複與不同步。
2. 快速看板「未解決問題」段該怎麼判斷起訖？
   - Codex 建議：先用下一個一級欄位作為結束點，例如遇到 `最後更新時間：` 就停止；如果找不到結束點，就讀到全文結尾。

請審閱本文件，並回答 §6 開放問題，再進入第 2 階段。
